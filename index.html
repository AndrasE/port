Sections

As you can see on the site there are slides coming right to left for each link you click on. These are sections, although it looks that they all different components of the page but they are really not!
Just imagine a standard webpage as a roll of paper. As you scroll down you go through different segments of the same paper. Now imagine this roll to be cut up for a given dimensions(sections) and start piling up these "card/sections" on top of each other. They all exist the same time, in front of you but most of them gets hidden away by the first card. As the new section slides in, the previous one stayes in the back. Each slides will have qualities such "hidden", "back-section", "active" as and when required.
When I finished the initial project, I realized there is an issue while loading the page. The last slide that loads will be visible for a second and kind of ruining the inital animation. Easy fix would be to add and empty black section (no characthers pictures or anything else) as the last card. When the DOM (document object model) gets loaded it will be the last card on top. Home section is "active" so will slide over it. Looks pretty neat. The problem with this solution is that as the DOM grows bigger and starts running the project NOT only locally, it will require more time for the DOM to load in, so the weird trace of the second flashing up will come back. Even though we are quick, but to pile the cards on top of each other as fast that noone will be able to notice anything is more fast than we the speed possibly can do. Not to mention that these pack of cards getting bulkier!
Okay the next solution to the problem was to set all not needed sections to be hidden until the DOM loaded. That worked again for a while, but every now and then, it seems the issue came back with a flash. The reason being is that the DOM loads first (line by line) and styling is after. You start putting the cards on top of each other, but you haven`t yet learnt the trick to hide what you are doing. Its becouse DOM is partially built by the time the CSS (which contians this trick) is loaded.. I realised I could do better.
Rather than leave a blank section as the last slide I should make it the first and add it as "back-section" right away. Like this, when the HTML and right after the CSS starts loading it will eliminate the issue. By the time it finishes the load the whole DOM (put the cards down) the back-section is already set (trick learnt) and the rest of the page would not be visible. Trickery! Altough it works this small/medium sized webpage, it wouldn`t be recomended to use this approach to anything significant. Either need to have the webpage component based (React) or an intersection observer needs to be set up. But I come back to this later on.

Jpeg vs. Webp

Out of interest, I wanted to see how else I can make the page load faster. I have around 5-6 pictures added to the project. By checking out in the network tab in dev mode I saw that each(Jpeg) takes around 600-700ms to load. Webp images have around 15-30% less data than Jpeg as it compresses more. But I also read on Stackoverflow that they are harder to decode, so the gain is not as significant. This answer was about 4 years ago, so I wasn't sure if it still stands. I had to make my own testrun for it, just wanted to know.
My original Jpegs weren`t anything big to start with, but making them as a Webp the size got reduced even further. I compared the page load time using Jpeg vs Webp. What I found is that it means a small improvement in my case. I talk around 0.0x seconds worth of improvement. Although I didn't gain anything much, it's clear to see I would have more and bigger resolution (larger) images it would make a difference. I just thought I share my findings after I invested in researching and test the topic.

Lazy-loading

Welcome to the realm of Lazy-loading! If you are unfamiliar with the term: "Lazy-loading is the practice of delaying load or initialization of resources or objects until they're actually needed to improve performance and save system resources."
Just imagine you scrolling through facebook/instagram, imagine that at the initial start of the app it would need to load in all the content for you. Would be quite a janke ride! Furthermore you might just open the site for a second or two, but it's taking over all of your bandwidth and loading all the content even if you won`t need them later on.
A solution exists! Many sites use an intersection observer. It's basically passing the message to the app what section of the site the user is currently "consuming". You can tell it "if the viewport is less than 500pixels away", start loading in the images, videos etc. If you never reach that far on the site, it shall never load. A win-win-win solution.
The reason I dive into this subject is because as I mentioned above I wanted to keep this project light-weight as possible. As soon as I embed a few YouTube videos, it went cra-cra in the network tab. YouTube/Google started to bombard the site with their scripts, figuring out what Ads to load and of course load the videos in. Remember, all sections are existing at the same time in this project. You might not look at the section where the video got embedded, but it will still load up.
I could use an intersection observer, but I didn't want to add an external library for this, especially if my only concern was the embedded videos. I went to find different solutions for this but I ended up creating my own one. Here is the work of others:

Solutions:   Example-1   Example-2

I liked the idea, but if you try it out, you can see that somehow is not really elegant. Many times the YouTube autoplay won't even work, so you basically need to click twice for the video to start. I also didn't like that a custom icon has to be used on top of the pictures.. Anyhow, here is my take on the subject:
Before I dive into this, we need to know what is Iframe. Iframe is (it is in its name) a frame which lets you embed another HTML page into the current one. It has its own properties like style, history and document. It`s not interact with the parent HTML. So what the guys were doing on that two example is that they put a image into the iframe with some icon on top as a play button using srcdoc property.
This lets you pretty much put the whole page as an inline element within the iframe. Anyways, let's not go too deep. There is a picture once it gets clicked will start to play the YouTube video as long autoplay works..
As I mentioned, I didn't like that it looks different and you need to double click a lot of times, but my main concern was that I would have to put a lot of lengthy inner styling in my HTML and I really like the clean look of the code as it was. I started to develop an idea to move everything to the JS to later be injected into the iframe when needed. But I still didn't like that there is a lot of HTML and CSS in my JS. And playing around hit me the simplest idea.
There is something called innerHTML. This can be set. I've done it before! What I did is that I prepped an empty div which will be injected using this innerHTML. What happens when the page loads, we have empty divs(containers if you like). But when you click on the whatever project it will call a function which will set the innerHTML to the iframe we need. It works as magic, but as YouTube is not messing around and right away starts to bombard you with the video and Ads so on, messed up my animation. What I did was I wrote another function with a loading gif image. When you click on the project, it's setting that image, but after 0.5s it's resetting the iframe again, now to the actual video. No double click or wierd icons and my code remained clean!
One thing to know that after you click away I also had to write a function as you may not on that section now, but youtube would just continue playing in the backgroubd and steal your bandwidth. So when you click away, the iframe will be set back to the gif I just mentioned. Pretty nice solution, I`m proud of it, it's my child!
